# GitHub 项目微信公众号文章生成 Skill

## 用户配置

在使用前，请确认以下配置（首次使用时设置，后续自动沿用）：

- **公众号名称**：`JY的拾光杂货铺`（替换为你的公众号名称）
- **默认受众**：对 AI 和效率工具感兴趣的普通用户
- **封面尺寸**：900×383（微信公众号封面标准 2.35:1）

---

## 定位与目标

本 skill 专为「GitHub 热门项目」类微信公众号内容创作设计，核心价值：
- 将技术性的 GitHub 项目转化为**面向普通读者**的中文科普内容
- 提炼项目的**实用价值**和**使用场景**，而非堆砌技术参数
- 生成符合微信平台调性的**高打开率标题**和**强互动文末 CTA**
- 输出内容直接可用于公众号发布，无需二次大改
- 文章生成后**自动推送到公众号草稿箱**，用户只需去后台确认发布

---

## 文章四段式框架（核心内容结构）

**这是本 skill 的内容骨架。** 每篇 GitHub 热门项目公众号文章，读者在不同阅读阶段的体验应该是层层递进的——看标题想炫耀、看开头被吸引、看过程学技术、看结尾受鼓舞。

### 第一段：标题即「炫耀帖」

读者看到标题的第一反应："这也太牛了吧！"

**强制要求：**
- 标题必须包含**具体数字**（Star 数、用户量、增速）或**行业地位**（第一、最大、最快、最轻量）
- 制造"不点进去就亏了"的紧迫感，但**禁止夸张词**（震惊、强烈推荐、全网首发等）
- 数字必须有时间戳锚定（如"截至 2026 年 6 月"）

**示例思路：**
- "148k Star，Airbnb 用了 8 年的 JS 编码规范"
- "109k Star 之后，Ray 正在吃掉 AI 计算的底层"
- "SQLMap 不是工具，是 18 年来每个渗透测试者的信仰"

**与风格矩阵的映射：**

| 风格 | 标题数字冲击方式 | 典型句式 |
|------|----------------|---------|
| `FUTURIST` | 56px 巨型数字 + "XX 之后" | "[数字] Star 之后，[项目名]正在改写规则" |
| `PRAGMATIC` | 红色对比卡（过去 vs 现在） | "[数字] Star，比 [旧方案]快 [X] 倍" |
| `ARCHITECT` | 深灰底白字规格卡 | "[数字] Star 的 [领域] 标准制定者" |
| `SCHOLAR` | 42px 研究焦点数字 | "被引 [N] 次的 [领域] 基石" |
| `MENTOR` | 时间线数字递增 | "从 [N] 到 [N×10] Star 的 [N] 年" |
| `EXPLORER` | emoji 标签云 + 数字 | "发现一个 [数字] Star 的宝藏" |
| `COMPANION` | 28px emoji + 轻量数据 | "[数字] Star 的 [类别] 神器" |

### 第二段：开头即「故事帖」

读者读完开头 3 段后的感受："原来是因为这样才诞生的"

**强制要求：**
- 以**故事驱动**开头：项目因什么痛点诞生？作者遇到了什么麻烦？最初只是一个什么场景下的尝试？
- 建立读者共情：让读者觉得"我也遇到过这个问题！"
- **禁止直接甩功能列表**——必须先用故事铺垫情境，让读者带入后再展开技术

**每种风格的钩子策略：**

| 风格 | 钩子策略 | 故事角度 |
|------|---------|---------|
| `SCHOLAR` | 学术困境 → 突破 | "XX 年，一群研究员在 [场景] 中发现……" |
| `EXPLORER` | 好奇心驱动 | "某个深夜，一个程序员随手搜到了一个 300 Star 的小项目……" |
| `PRAGMATIC` | 效率折磨 → 解决方案 | "每次 [操作] 要花 [N] 分钟，直到有人忍无可忍……" |
| `MENTOR` | 作者个人故事 | "[作者名] 在 [年份] 还只是一个 [身份]……" |
| `FUTURIST` | 未来场景预演 | "想象一下：[N] 年后的 [场景] 会是什么样？" |
| `ARCHITECT` | 行业缺口发现 | "整个 [领域] 一直缺一个 [能力] 的标准实现……" |
| `COMPANION` | 朋友安利式 | "最近我发现了一个工具，第一反应是：怎么不早点出来！" |

### 第三段：正文即「技术帖」

读者浏览正文时的感受："原来这么强"

**强制要求：**
- 按 **3-5 个核心功能**分层展开，每个功能配**使用场景**（不是功能罗列，是"在 XX 场景下你可以用它来做什么"）
- **数据说话**：Star 数、性能对比 benchmark、真实用户反馈引用
- **代码块/命令行**展示上手门槛（PRAGMATIC/ARCHITECT/EXPLORER/COMPANION 风格强制有）
- **必须有至少一处使用限制说明**（保持客观，如仅支持某语言、需要 GPU、实验性功能等）

**段落节奏规则：**
- 每个 h2 章节不超过 3 段文字（避免大段纯文字）
- 每 2-3 段文字之间至少插入 1 个视觉锚点（表格/引用块/图片/代码块）
- 功能介绍的顺序：从"最常用"到"最炫酷"（或用"基础→进阶→高级"递进）

### 第四段：结尾即「励志帖」

读者看到结尾时的情绪："开源的力量真了不起"

**强制要求：**
- **升华主题**：从单个项目上升到**行业趋势、开源精神、技术民主化**
- **传递积极情绪**：让读者觉得"我也可以参与"或"技术正在让世界变好"
- **转发金句 + CTA 投票** → 引导读者行动（转发/讨论/上手试用）
- 结尾语气：温暖的、向上的，非说教

**升华方向参考（按项目类型）：**

| 项目类型 | 升华角度 | 示例金句方向 |
|---------|---------|------------|
| 个人开发者的工具 | 一个人的坚持改变数十万开发者 | "一个程序员的一个周末项目，改变了 XX 的工作方式" |
| 大厂开源的框架 | 开放标准的力量 | "当 [公司名] 把这行代码开源时，整个行业被重新定义" |
| 学术研究工具 | 知识平权 | "那些曾经只在论文里的算法，现在任何人一行命令就能用" |
| 效率工具 | 时间复利 | "每天省下的 10 分钟，乘以 [N] 万用户，是改变世界的时间" |
| AI 前沿项目 | 未来已来 | "曾经只存在于科幻电影里的场景，正在被这个项目变成现实" |

### 四段式与 7 种风格的一键映射

```
标题（炫耀帖）→ 风格专属的数字冲击方式（见上表）
      ↓
开头（故事帖）→ 风格专属的钩子策略（见上表）
      ↓
正文（技术帖）→ 风格专属的布局模板（第二步风格-布局配对表）
      ↓
结尾（励志帖）→ 风格专属的金句模板 + CTA 模板（见 references/styles.md）
```

### 四段式验证清单（每篇文章必须打勾）

- [ ] **炫耀帖**：标题含具体数字或行业地位定位 ✓
- [ ] **故事帖**：开头 3 段讲了痛点/起源/作者故事，非功能列表 ✓
- [ ] **技术帖**：正文含 3-5 个功能 + 使用场景 + 至少 1 处限制说明 ✓
- [ ] **励志帖**：结尾有趋势升华 + 金句 + CTA（非"欢迎关注"） ✓
- [ ] **四段连贯**：从故事自然过渡到技术，从技术自然升华到励志 ✓

## 输入要求

用户需提供以下任意一项或多项：
- GitHub 项目 URL（如 `https://github.com/owner/repo`）
- 项目名称
- 项目简要描述
- （可选）目标受众偏好：技术开发者 / 产品经理 / 效率达人 / 学生 / 通用

若未提供目标受众，默认面向「对 AI 和效率工具感兴趣的普通用户」。

## 执行流程

### 第一步：项目信息收集

**若用户提供了 GitHub URL**，使用 WebFetch 抓取以下信息：
1. 项目主页（README 首屏）
2. Star 数、Fork 数、License 类型
3. 项目语言、最近更新时间
4. Release/Tag 页（了解版本成熟度）

**若只有项目名或描述**，使用 WebSearch 搜索：
- `site:github.com [项目名]` 找到官方仓库
- `[项目名] 教程 OR 使用场景 OR 案例` 找中文参考内容

**关键信息清单（必须获取）：**
- [ ] 项目用一句话能做什么
- [ ] 解决了什么痛点 / 比现有方案好在哪里
- [ ] 面向什么人群（开发者/设计师/普通用户/所有人）
- [ ] 核心功能 3-5 个
- [ ] 是否有在线体验 / Demo / 安装门槛高不高
- [ ] Star 数量级（千 / 万 / 十万+）— **必须标注"截至 202X年X月"**
- [ ] 是否开源免费
- [ ] 已知缺陷/限制（尤其是早期项目或实验性功能）

**真实性验证清单（必须打勾）：**
- [ ] **Star 数加时间戳**：写成"截至 2026 年 6 月，Star 数 XX 万+"，不写裸数字（半年后 Star 数会变，裸数字会让老文章显得过时）
- [ ] **项目限制标注**：如项目有明确短板（仅支持英文、需要 GPU、AI 回答可能出错），必须在文章中提到至少一处限制说明
- [ ] **用户反馈引用**：搜索项目的 Issue/Discussion/Hacker News/Reddit 中真实用户的评价，至少引用 1 条（改写为人话，不逐字翻译），让文章有"真人用过"的质感
- [ ] **同类对比准确**：如果文章中提到"比 XX 更好/更快"，必须确认该对比在项目 README 或独立 benchmark 中有据可查
- [ ] **价格/付费说明**：如项目有付费版，必须标注"核心功能免费，XX 功能需付费"

#### 风格归入判断流程

拿到项目后，按以下优先级判定风格：
1. **看项目本质**（最重要）：它是解决什么问题的？
   - 是研究/学习/知识管理？→ `SCHOLAR` 或 `MENTOR`
   - 是数据聚合/趋势洞察？→ `EXPLORER`
   - 是基础工具/国产开源/性能相关？→ `PRAGMATIC`
   - 是框架/协议/平台？→ `ARCHITECT`
   - 是 AI 前沿/机器人/世界模型？→ `FUTURIST`
   - 是个人工具/本地应用？→ `COMPANION`
2. **看项目背景**（微调）：作者有传奇故事？→ 可偏向 `MENTOR`；大厂出品定义标准？→ 可偏向 `ARCHITECT`；中国团队/国产之光？→ `PRAGMATIC` 优先
3. **看 Star 数**（辅助判断）：10k 以下小众项目 → `COMPANION` 或 `EXPLORER` 更合适

### 第三步：生成文章（⚠️ 强制执行布局模板，不可跳过）

**这是整个 skill 最关键的步骤。第 7 批 30 篇文章的 audit 发现：布局模板使用率为 0%，导致所有文章视觉同质化。此步骤现在强制执行。**

#### 3.0 执行规则（不可违反）

1. **必须打开对应的布局模板文件**（`references/layouts/{layout}.md`）
2. **必须逐段复制模板中的 HTML 结构**，替换 `{{占位符}}` 为实际内容
3. **禁止自创结构**——即使你觉得某个布局元素"不适合这个项目"，也必须保留其结构，只替换内容
4. **禁止混用多个布局模板**——一篇文章从头到尾只用一个布局
5. **代码块必须存在**——PRAGMATIC/ARCHITECT/EXPLORER/COMPANION 风格要求至少 1 个 `<pre>` 代码块（安装命令或示例代码），不可省略（2026-06-11 audit：30 篇文章 0 篇有 `<pre>`，现强制执行）
6. **封面图必须存在**——每篇文章必须有 Base64 PNG 封面图 `<img src="data:image/png;base64,...">`（来自 GitHub OG），无封面 = 废稿（2026-06-11 audit：25/30 篇缺封面，现强制执行）
7. **分割线必须存在**——每篇文章至少 2 处 `<hr>`（金句前 + 点赞引导前），长文（>1500字）需 3-4 处
8. **`border-radius` 必须为零**——全文 grep `border-radius` 结果必须为空（2026-06-11 audit：27/30 篇违规使用）
9. **文件必须以 `<!DOCTYPE html>` 开头**——禁止 `<section>` 包裹全文这种偷懒写法

#### 3.1 7 种布局签名视觉速查

| 风格 | 布局 | 签名视觉（3秒可辨认） | 炫酷元素 | 代码块 |
|------|------|---------------------|---------|--------|
| SCHOLAR | THESIS | 深蓝底白字「研究焦点块」+ 大数字 | 42px 白字核心数据 | ❌ 无（段落描述） |
| EXPLORER | MAP | emoji 标签云「探险兴趣点」+ 青色虚线 | 3 列 26px emoji 标记 | ✅ 浅色底 |
| PRAGMATIC | DASHBOARD | 2×2 大数字网格 + 红灰撕裂对比卡 | 28px 红色数字 + ❌/✅ 对比 | ✅ 深色终端 `#1e1b4b` |
| MENTOR | LETTER | 渐变色「故事时间线卡」+ 书籍目录 | 3 行渐进紫底里程碑 | ✅ 浅色底 |
| FUTURIST | MANIFESTO | 56px 巨型数字 + 深蓝底认知颠覆卡 | 56px 白字 + 左灰右蓝对比 | ❌ 无 |
| ARCHITECT | BLUEPRINT | 深灰底「规格说明书卡」+ 三层架构 | `#0f172a` 底白字 + 🔷 层级 | ✅ 最深色 `#0f172a` |
| COMPANION | CHAT | emoji「第一印象反应卡」+ 聊天子弹 | 28px emoji + 绿色 ✨ 列表 | ✅ 浅色底 |

#### 3.2 生成流程（必须逐步执行）

```
步骤 A：打开 references/layouts/{配对的布局}.md
步骤 B：从模板中复制 h1 → 结尾的完整 HTML 骨架
步骤 C：将 {{文章标题}} {{STARS}} {{BASE64_COVER}} 等占位符替换为实际值
步骤 D：为 {{钩子段落}} {{功能描述}} {{金句}} {{CTA}} 填入从 styles.md 对应风格选定的内容
步骤 E：按「四段式框架」检查内容结构：
    ① 标题是否够"炫耀"（数字/地位）
    ② 开头是否讲了故事（痛点/起源）
    ③ 正文是否展示了技术（功能+场景+数据）
    ④ 结尾是否做了升华（趋势+金句+行动号召）
```

**关于封面图（强制规则）：**
- 封面图**必须**使用 **GitHub 真实 OpenGraph 图片**：`<img src="data:image/png;base64,...">`
- GitHub OG 图片 = GitHub 自动生成的项目卡片（项目名 + 描述 + Star 数 + 语言标签），是最权威、最干净的封面来源
- **禁止**使用 Pillow/AI 自绘封面、禁止通用渐变底色封面
- **禁止**使用仓库自定义上传的 `repository-images.githubusercontent.com` 图片（多为模糊截图）
- 下载脚本参考：项目内 `download_github_og.py`

**GitHub OG 封面获取流程（强制）：**

1. **获取 og:image URL**：
   ```bash
   curl -sL -A "Mozilla/5.0" "https://github.com/{owner}/{repo}" \
     | grep -o 'property="og:image" content="[^"]*"'
   ```

2. **判断 URL 类型**：
   - `opengraph.githubassets.com/{hash}/owner/repo` → 直接下载 ✅（GitHub 自动生成的干净卡片）
   - `repository-images.githubusercontent.com/...` → **丢弃** ❌（仓库自定义模糊截图）

3. **如果仓库用了自定义图片，强制用 opengraph URL**：
   用任意有效 hash 构造：`https://opengraph.githubassets.com/{任意有效hash}/{owner}/{repo}`
   （GitHub 不验证 hash，只要格式正确就返回自动生成卡片）

4. **下载图片**：
   ```bash
   curl -sL --noproxy "*" -A "Mozilla/5.0" --max-time 60 \
     -o cover.png "https://opengraph.githubassets.com/{hash}/{owner}/{repo}"
   ```

5. **转 base64 嵌入**：
   ```python
   import base64
   with open('cover.png', 'rb') as f:
       b64 = base64.b64encode(f.read()).decode('ascii')
   data_uri = f'data:image/png;base64,{b64}'
   ```

6. **替换 HTML 封面**（找到第一个 `data:image/` 的 img 标签替换）

**已知仓库迁移（必须用新地址）**：
- `comfyanonymous/ComfyUI` → `Comfy-Org/ComfyUI`
- `ggerganov/llama.cpp` → `ggml-org/llama.cpp`

**关于 `<pre>` 代码块（强制规则）：**
- PRAGMATIC 风格：必含安装命令（如 `npm install bun` / `pip install`）
- ARCHITECT 风格：必含架构级示例代码
- EXPLORER 风格：必含快速上手命令
- COMPANION 风格：必含安装/启动命令
- FUTURIST/SCHOLAR/MENTOR：可免，但如有代码示例必须用浅色底
- `<pre>` 内命令不超过 3 行，超过用户会跳过
- 配色严格遵守 5.6 强制映射表

**关于 `<hr>` 分割线（强制规则）：**
- 至少 2 处：① 转发金句前（"正文结束"信号）② 点赞引导前
- 分割线样式：`<hr style="border:0;border-top:1px solid #e5e5e5;margin:24px 0;">`
- 长文（>1500字）建议 3-4 处分割线

**关于 `border-radius`（禁令）：**
- 全文 `grep border-radius` 结果必须为 0

**关于 `<!DOCTYPE html>`（强制规则）：**
- 每个 HTML 文件首行必须是 `<!DOCTYPE html>`

- [ ] ⚠️ **正文图片必须上传到微信 CDN**：所有 `<img>` 的 `src` 不得为 Base64 Data URI，必须使用 `mmbiz.qpic.cn` 域名。推送前需通过 `cgi-bin/media/uploadimg` 将图片逐个上传并替换 HTML 中的 src（Base64 Data URI 在微信 API 推送时会被丢弃，图片不显示）

#### 推送结果输出格式

```
✅ 草稿发布成功！

📝 文章信息
- 标题：{title}
- 摘要：{digest}
- 草稿 media_id：{DRAFT_MEDIA_ID}

📌 前往公众号后台查看并发布：
   https://mp.weixin.qq.com （登录后点击「内容管理」→「草稿箱」）

⚠️ 检查要点：
1. 封面图是否正确显示
2. 排版在手机预览中是否正常
3. 标题和摘要是否合适
4. 确认无误后点击「发布」
```

#### 推送失败的兜底处理

如果推送失败（token 过期、网络错误、API 报错等）：
1. **不要静默失败**——必须向用户明确报告错误原因
2. 提供**手动发布路径**：告知用户文章 HTML 文件已保存，可以手动复制到微信编辑器
3. 常见错误的处理建议：
   - Token 无效 → 检查 WECHAT_APPID / WECHAT_SECRET 是否正确
   - 标题过长 → 自动截断并提示用户审查
   - 图片上传失败 → 检查封面图格式和大小
   - API 频率限制 → 等待后重试或提示用户手动发布

本 skill **依赖** `mp-draft-push` skill。调用方式：
1. 使用 `Skill` 工具加载 `mp-draft-push`
2. 将准备好的参数传入
3. `mp-draft-push` 负责获取 access_token、上传封面图、创建草稿

**无需在本 skill 中重复实现微信 API 调用逻辑**——`mp-draft-push` 已封装完整的发布流程。

## ⚠️ 微信编辑器兼容规则（极其重要）

微信公众号编辑器是一个**经过严格过滤的富文本编辑器**，不是完整的浏览器。粘贴 HTML 时，大部分 CSS 会被丢弃。

### 微信会自动删除的样式

| 写法 | 结果 | 说明 |
|------|------|------|
| `<div class="xxx">` | ❌ class 全部丢掉 | 永远不要依赖 class |
| `<div style="background:...">` | ❌ div 的背景色丢掉 | div 的任何视觉样式都靠不住 |
| `<style>` 标签 | ❌ 整个丢弃 | 不能放任何 CSS 类定义 |
| `display:flex` | ❌ flex 布局全丢 | 只能用 table 布局 |

### 微信会保留的写法

| 写法 | 结果 | 说明 |
|------|------|------|
| `<table>` + `bgcolor` 或 `style="background:..."` | ✅ 保留 | **做卡片的首选方案** |
| `<blockquote style="border-left:...">` | ✅ 保留 | 做高亮引用块 |
| `<h2 style="...">` | ✅ 保留 | 标题样式保留 |
| `<strong style="color:...">` | ✅ 保留 | 重点文字颜色保留 |
| `<img>` | ✅ 保留 | 图片 |

### 强制转换规则

生成 HTML 时必须遵守以下规则：
1. **禁止使用 `<div>` 做卡片** → 一律改为 `<table width="100%" cellpadding="14" cellspacing="0" border="0" style="background:#xxx;">` 包裹内容
2. **禁止使用 class** → 所有样式写为内联 `style="..."`
3. **禁止使用 `<style>` 标签** → 全文零 CSS 类
4. **禁止使用渐变、圆角、阴影** → 微信不支持，写了也是白写。**特别强调：`border-radius` 在微信内置浏览器中渲染不一致，部分版本完全丢弃，必须禁用。** 替代方案：用 `border-left:3px solid {{主题色}}` 做视觉区分，用 `padding` 做呼吸空间
5. **禁止使用 flex/grid 布局** → 用 table 替代

## 📷 图片处理规范（极其重要）

### 图片粘贴方案：Base64 内嵌

微信公众号编辑器**拒绝一切外部图片入链**（本地路径、远程 URL 全部拦截），但**支持 Base64 Data URI**：
- `<img src="data:image/png;base64,iVBOR...">` → ✅ 浏览器复制 → 粘贴到微信 → 图片随 HTML 一起走

**因此：HTML 正文中的图片一律使用 Base64 Data URI 格式，不依赖任何外部文件。**

### 图片处理工作流

**封面图（OG 下载）**——使用 `download_github_og.py`（完整流程见第六步 5.7 节）：
```bash
python download_github_og.py --owner <owner> --repo <repo> --output cover.png
```

**正文配图（README 截图/Demo）**——使用 `scripts/process_images.py`：
```bash
python scripts/process_images.py --project-url <GitHub URL> --output-dir <项目名>-images
```

脚本自动完成：
1. **收集图片**：从 GitHub README 提取图片链接并下载
2. **压缩 GIF**：隔帧取样 + 缩放 + 调色板降色，目标每张 <5MB
3. **Base64 嵌入**：生成 `<img src="data:image/xxx;base64,...">` 并输出到 HTML

### 封面图规范

> ⚠️ **封面图强制规则已统一收口至第六步 5.7 节「关于封面图（强制规则）」**，此处不再重复。此处仅说明文件存储约定。

封面图**不能**通过 Base64 粘贴（微信封面是独立上传入口），需手动上传。封面图文件存入 `{项目名}-images/cover.png`。

**唯一来源**：GitHub OpenGraph 自动生成卡片（`opengraph.githubassets.com`），详见第六步 5.7 节的完整获取流程。

**禁止行为**：
- ❌ 使用 README 截图/Demo 动图做封面（信息密度低、视觉不统一）
- ❌ 使用项目 Logo 做封面（尺寸不可控、品牌感弱）
- ❌ 使用 Pillow/AI 自绘封面（风格割裂、可信度低）
- ❌ 使用 `repository-images.githubusercontent.com` 模糊截图

## 📸 配图传播策略（⭐ 3.5.1 新增）

> 配图不是填充物——它是文章传播的"燃油"。一张对的图胜过三段文字，一张错的图让读者划走。

### 核心原则：配图是"分享冲动"的触发器

微信公众号文章传播的本质是**社交分享**。读者不会因为"这篇文章写得不错"而转发，他们转发是因为看到了某样东西产生情绪冲动——"这个好酷""原来还能这样""太强了"。而**图片比文字快 6 万倍触发大脑情绪反应**。

因此，配图策略的核心问题是：**这张图能不能让读者产生分享冲动？**

### 配图传播力矩阵

不同类型的图片天然携带不同的传播势能。以下是 GitHub 项目中可获取的图片类型与传播力评级：

| 图片类型 | 传播力 | 读者心理反应 | 获取来源 | 适用风格 |
|---------|--------|-------------|---------|---------|
| **Before/After 对比图** | ⭐⭐⭐⭐⭐ | "这也太强了吧！"——直接引发转发 | README 演示截图、Issues 用户反馈 | PRAGMATIC、ARCHITECT |
| **性能 Benchmark 图表** | ⭐⭐⭐⭐⭐ | "数据不会骗人"——理性说服转社交货币 | 项目 Wiki、README benchmark | SCHOLAR、PRAGMATIC |
| **Demo/Screencast GIF** | ⭐⭐⭐⭐ | "原来长这样！"——满足好奇心 | README、官网 landing page | EXPLORER、COMPANION |
| **终端命令行演示截图** | ⭐⭐⭐⭐ | "这就是我想要的东西"——极客认同 | README 安装/使用章节 | ARCHITECT、FUTURIST |
| **架构/流程图** | ⭐⭐⭐ | "这东西想得很清楚"——建立信任 | README、项目文档 | ARCHITECT、SCHOLAR |
| **功能界面截图** | ⭐⭐⭐ | "看起来很强大"——功能感知 | 官方文档截图 | 所有风格 |
| **Logo/品牌图** | ⭐⭐ | 仅品牌识别，无传播触发 | GitHub Repo、官网 | 仅尾缀使用 |
| **代码截图（纯代码）** | ⭐ | "好长……"——负向驱离 | README 代码 | ❌ 避免，改用 `<pre>` 代码块 |

### 四段式配图节奏

配图不是均匀撒在文章各处——不同段落需要不同类型的图片来强化对应的阅读体验：

```
炫耀帖段（标题 + 封面区）
  配图策略：1 张就够了
  首选：GitHub OG 封面图（Star 数 + 项目名，信息密度最高）
  禁忌：堆叠多张 Logo 或无关背景图
  ↓
故事帖段（开头 3 段）
  配图策略：0-1 张，可选
  可选：项目早期版本截图、作者 Twitter/GitHub 截图、社区讨论热图
  目的：强化"这项目有故事"的真实感，非技术展示
  禁忌：此处放功能截图会打断叙事节奏
  ↓
技术帖段（正文核心）
  配图策略：3-5 张，重点投放区
  必选 1：项目核心 Demo GIF 或主界面截图（让读者"看到"它能做什么）
  必选 2：Before/After 对比图 或 Benchmark 图表（传播触发器）
  可选：架构图、命令行演示、功能拆解截图
  原则：每张图配 ≤ 2 句说明文字，图比字更有力
  ↓
励志帖段（结尾升华）
  配图策略：0-1 张，可选
  可选：社区贡献者头像墙、项目 Star 增长曲线、未来路线图
  目的：强化"开源力量"的情绪共鸣
  禁忌：此处放技术图表会破坏升华情绪
```

### 配图选择的三步决策法

面对一个项目，按以下顺序决定配图：

```
步骤 1：有没有 Before/After 对比图？
  → 有 → 必选，这是整篇文章的传播引爆点
  → 无 → 进入步骤 2

步骤 2：有没有 Demo GIF 或动图展示核心功能？
  → 有 → 必选，放在技术帖段第一张
  → 无 → 截图静态界面，加箭头标注关键功能区

步骤 3：有没有 Benchmark 或性能对比数据？
  → 有 → 做成可视化表格（数据卡）+ 可选图表
  → 无 → 用 Star 数 + 用户反馈引用替代数据说服
```

### 配图质量红线

**所有用于文章的配图必须通过以下检查：**

- [ ] **截图完整**：不裁剪到一半的功能区，不包含无关的浏览器标签页/任务栏
- [ ] **文字可读**：截图中的代码/菜单文字在手机上缩放后仍然可读（原图宽度 ≥ 800px）
- [ ] **GIF ≤ 5MB**：微信对 GIF 文件大小敏感，超过 5MB 加载缓慢影响体验。如需压缩，调用 `scripts/process_images.py` 自动处理
- [ ] **无外部依赖**：正文中的所有图片已转为 Base64 Data URI，不依赖外部 URL
- [ ] **标注清晰**：对比图左侧/右侧有明确标注（如 "❌ 之前" / "✅ 之后"），不靠读者自己猜测
- [ ] **风格一致**：同一篇文章的所有截图统一窗口主题（暗色/亮色二选一，不混用）
- [ ] **无敏感信息**：截图中不含作者个人邮箱、API Key、内部 IP 等敏感信息

### 配图数量与节奏控制

| 文章总字数 | 建议配图数 | 配图密度 |
|-----------|-----------|---------|
| 800-1200 字 | 3-5 张 | 约每 250 字 1 张 |
| 1200-1800 字 | 5-7 张 | 约每 250 字 1 张 |
| 1800-2500 字 | 6-8 张 | 约每 300 字 1 张 |

**上限**：单篇文章正文配图不超过 8 张（不含封面）。超过 8 张会导致 Base64 HTML 文件过大、微信编辑器卡顿。

**节奏感**：配图应是"呼吸"——密集（技术帖）→ 舒展（励志帖），而非均匀撒落。技术帖段可以连续 2-3 张图紧挨着，励志帖段则完全不放图。

### 传播型配图的反面案例

以下是会**杀死**文章传播力的配图错误：

| ❌ 错误做法 | 读者反应 | 正确做法 |
|------------|---------|---------|
| 用 Logo 图替代功能截图 | "不知道它到底能做什么" | Logo 只放尾缀，功能区用真实截图 |
| 截图包含大量无关 UI 元素 | "好乱，看不懂" | 裁剪到核心功能区，加箭头标注 |
| 多张截图风格不一致（黑底+白底混搭） | "不专业" | 统一窗口主题，暗色截图转亮色或反之 |
| 没有 After/Before 对比 | "没什么震撼的感觉" | 至少找 1 组对比：vs 同类工具 或 使用前 vs 使用后 |
| GIF 过大导致加载卡顿 | 直接划走 | ≤ 5MB，必要时降帧率到 10fps |
| 截图包含 "Loading..." 或错误提示 | "有 Bug 吧" | 等界面完全加载后再截图 |
| 代码截图代替 `pre` 代码块 | 文字模糊不可选中 | 代码一律用 `<pre>` 代码块，截图只展示 UI 效果 |

## 特殊情况处理

### 项目信息不足时
如果 GitHub 项目内容较少（新项目、README 简单），补充以下动作：
1. 搜索该项目的 Issue、Discussion 了解用户反馈
2. 搜索该项目创作者的介绍/采访内容
3. 搜索同类项目对比，从差异化角度切入
4. 在文章中适当标注"目前该项目仍在早期阶段，功能持续完善中"

### 纯技术开发者工具
若项目受众主要是程序员（如 CLI 工具、库、框架），调整策略：
- 标题偏向"程序员专属"、"开发神器"
- 功能介绍可以适当加入代码片段（用代码块格式，不超过 10 行）
- 文末 CTA 改为"这个工具你用过吗？你的使用场景是什么？"

### 企业级 / 付费项目
若项目有商业付费版，必须：
- 明确说明开源版的功能范围
- 不做虚假宣传"完全免费"
- 可以说"核心功能完全开源免费，企业版提供更多高级功能"

## 🎨 排版美学原则

> **详细指南**：见 `references/layout-aesthetics.md`（包含完整的设计系统、案例对比、自查清单）

微信公众号文章本质上是**手机竖屏阅读**——读者在一块 375px 宽的屏幕上，用拇指滑动浏览。以下 7 条核心原则是文章的"视觉底层"。

### 原则 1：视觉层级（Visual Hierarchy）

读者扫读一篇文章只需 3-5 秒来决定是否继续读。号主必须在这几秒内建立清晰的视觉层级：

| 层级 | 元素 | 字号 | 颜色 | 作用 |
|------|------|------|------|------|
| L0 | 封面图 | — | — | 第一视觉冲击，决定点不点进来 |
| L1 | h1 主标题 | 24px | `#1a1a1a` | 进来看的第一件事：我在读什么？ |
| L2 | h2 章节标题 | 18-19px | `#1a1a1a` + 装饰 | 扫读锚点，翻到就知道这段讲什么 |
| L3 | 关键数据/金句 | 16-22px | 主题色 | 眼睛自然停留的数个字/一句话 |
| L4 | 正文段落 | 14-15px | `#333` | 详情阅读区 |
| L5 | 辅助信息 | 12-13px | `#999` | 星数、日期、引导语等 |

**核心规则**：任意相邻两个层级的字号差 ≥ 2px，否则层级模糊。

### 原则 2：色彩系统三色法则

每篇文章只使用 **3 种功能性颜色**：

| 颜色角色 | 用途 | 限制 |
|---------|------|------|
| **主题色** | h2 装饰、关键数字、CTA 按钮、投票文字 | 全文唯一，来自风格配色表 |
| **正文色** | 段落文字 | `#333`（固定，不用纯黑 `#000`） |
| **辅助灰** | 星数标签、日期、引导语 | `#666` / `#999`（固定） |

**禁止**：同一篇文章出现 2 种以上主题色混搭（如同时出现红色和蓝色强调）。

### 原则 3：间距节奏（Spacing Rhythm）

微信文章没有 CSS margin-collapse。所有间距靠手写 margin 控制。推荐使用 **4-12-20 节奏**：

```
小间距（元素内部）：4-8px    — 标签内、列表项间
中间距（同层元素）：12-18px   — 段落间、卡片间
大间距（章节分割）：20-32px   — h2 上下、分割线前后
```

h2 标题的 margin 必须**上大下小**（如 `margin:28px 0 14px`），形成"标题带领内容"的归属感。

### 原则 4：留白是设计

- 文章开头留白：h1 之后、封面图之前，至少 8px 间距
- 文章结尾留白：最后一段文字到"点赞引导"之间，至少 16px
- 表格卡片必须 padding ≥ 10px，文字不能贴边
- 分割线上下至少 20px

### 原则 5：移动端优先

- 段落宽度自适应（`width:100%` 或不用 width）
- 表格不要超过 4 列（5 列以上在手机上挤成面条）
- 图片 `max-width:100%`（防止撑破屏幕）
- 代码块字号 ≥ 13px（太小在手机上需要放大才能看）
- 链接文字不要写"点击这里"（手指不是鼠标），写"访问官网"等

### 原则 6：卡片设计规范

微信中做卡片只能用 `<table>`。卡片三大要素：

```
┌─────────────────────────────┐
│ 卡片标题（13px, #666）       │  ← 轻量标签
│                             │
│ 卡片核心内容（14-22px, 主题色 │  ← 视觉焦点
│ 或 #333, bold）              │
│                             │
│ 卡片辅助信息（12px, #999）    │  ← 补充说明
└─────────────────────────────┘
```

- 卡片背景色必须与页面背景有区分（至少 5% 亮度差）
- 同一页面卡片不超过 6 个（过多会让读者麻木）
- 数据卡片和功能卡片的背景色必须不同（分别用风格指定的数据卡底色和功能卡底色）

### 原则 7：视觉锚点（Visual Anchors）

每屏（约 500px 高度）至少放置 1 个视觉锚点，让读者有"抓手"：

| 锚点类型 | 适用场景 | 示例 |
|---------|---------|------|
| 数字锚点 | 数据型项目 | 超大字号 Star 数 |
| 颜色锚点 | 任何文章 | 彩色 `<strong>` 关键词 |
| 形状锚点 | 对比型内容 | ✅/⚠ 符号引导的清单 |
| 引用锚点 | 观点型内容 | `<blockquote>` 包裹的金句 |
| 图片锚点 | Demo 型项目 | 项目截图/动图 |

**禁止连续 3 段纯文字无任何视觉锚点**——读者的眼睛会迷失。

→ **完整设计系统、各风格视觉差异化策略、常见排版误区和自查清单**：见 `references/layout-aesthetics.md`

## 质量标准

一篇合格的文章应满足：
- 标题中有数字或具体场景，无"震惊/强烈推荐"等夸张词
- 正文中"你"的出现次数 > "用户"的出现次数
- 有至少一个真实使用场景的具体描述
- 文末有明确的互动引导（不是"欢迎关注"这种泛泛的）
- 全文没有超过 100 字的连续段落（在微信中不利于阅读）
- 对技术门槛有说明（告诉读者能不能轻松上手）
- **不同文章之间的配色方案、钩子句式、h2 模板、CTA 模板必须有明显差异**
- **不同文章之间的排版布局必须不同**：章节顺序、数据卡格式、功能展示方式、封面位置、间距密度、h2 装饰风格、分割线类型、CTA 格式等 8 个维度，每两篇文章至少有 4 个维度不同
- **每篇文章必须包含转发金句模块**（`<table>` 包裹的精炼观点金句 + "长按复制转发"引导）
- **Star 数必须标注时间戳**（"截至 202X 年 X 月"），方便读者判断信息时效性
- **每个项目至少提 1 个使用限制**（仅支持某语言、需要 GPU、实验性功能、付费版差异等）
- **⭐ 四段式框架完整**（3.5.0 新增）：标题有"炫耀"感、开头讲"故事"、正文展"技术"、结尾做"励志"升华，四段连贯不割裂
- **⭐ 自动推送完成**（3.5.0 新增）：文章通过全部自检后，自动调用 mp-draft-push 推送到公众号草稿箱，并输出后台直达链接
- **⭐ 配图传播策略达标**（3.5.1 新增）：正文至少 1 组对比图/Demo GIF + 配图质量通过全部 7 项检查 + 配图数量在 3-8 张范围内 + 每张配图 ≤ 5MB
